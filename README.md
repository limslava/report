## 🎯 Назначение системы

Система предназначена для ежедневного учета оперативных данных по логистическим направлениям, автоматического расчета плановых показателей, формирования отчетов в форматах PDF/Excel и рассылки результатов заинтересованным лицам по email.

**Основные пользователи:**
- Операторы направлений (5 направлений)
- Руководители направлений
- Топ-менеджмент
- Администраторы системы

**Направления деятельности:**
1. Контейнерные перевозки – Владивосток
2. Контейнерные перевозки – Москва
3. ЖД перевозки
4. Автовозы
5. Дополнительные услуги

## 🚀 Функциональные возможности

### Модуль ввода оперативных данных
- Excel-подобный интерфейс для каждого направления
- Автоматический расчет формул в реальном времени
- Валидация вводимых данных
- История изменений по каждой ячейке
- Автосохранение

### Модуль отчетности
- **Ежедневные отчеты** по направлениям с графиками выполнения плана
- **Операционный отчет (план с переносом)** с автоматическим расчетом плана с переносом
- Визуализация выполнения плана (%)
- Экспорт в Excel с сохранением формул
- В Excel-дашбордах показатель «Факт за месяц» скрыт (в API остается для расчетов)
- В Excel-дашбордах: `Δ ДЗ/КЗ` и `Ср. стоимость заявки` рассчитываются по формуле в backend

### Модуль email-рассылки
- Настройка расписания отправки (ежедневно, еженедельно, ежемесячно)
- Шаблоны писем для разных получателей
- Прикрепление отчетов в форматах PDF/Excel
- Лог отправленных писем
- Типы рассылок: СВ (ежедневный), Планирование v2 по сегменту, **СВ за месяц (итоговый)** (`monthly_final`)
- Надежность рассылок: ретраи при временных сбоях и защита от пропусков (в работе)

- Ежемесячный обзор по водителям и ТС (табличный вид, группировка по типам ТС) — в разработке

### Модуль управления доступом
- Регистрация по приглашению
- Ролевая модель (роль определяет доступ к сегментам/направлениям)
- Роли: admin, director, financer, manager_sales, head_sales, manager_*
- Восстановление пароля

### Бизнес-процесс согласования договоров
- Разделы `Согласование договоров`, `Мои договоры` и `Реестр договоров` для участников процесса.
- Менеджер отдела продаж (`manager_sales`) и руководитель отдела продаж (`head_sales`) выступают инициаторами: создают договоры, прикладывают документы, запускают согласование и работают со своими черновиками или редакциями.
- Инициатор не получает полномочий согласующего: поставить или изменить визу может только назначенный участник маршрута либо администратор.
- Маршрут согласования: для расходных договоров и доходных договоров с ПСР параллельно работают руководитель СБ, юрист, главный бухгалтер и финансовый директор; для доходных договоров без ПСР используется укороченный маршрут руководитель СБ → офис-менеджер.
- Согласующие могут выбирать решение, оставлять комментарии и прикреплять материалы к своей визе; комментарий обязателен при решении `Согласован с замечаниями`.
- Переход к офис-менеджеру происходит после факта визирования обязательных участников, независимо от типа визы.
- Вложения согласующих отображаются в ходе согласования и доступны участникам процесса для просмотра.
- Офис-менеджер получает задачу подготовить пакет на подпись, распечатать его и приложить скан подписанного экземпляра.
- Если подписанный экземпляр оказался у инициатора, он также может приложить скан вместо офис-менеджера; интерфейс показывает ему только компактное действие загрузки файла.
- Печатный пакет формируется в PDF в порядке: лист согласования, вложения согласующих, файлы договора.
- Файлы `.docx` доступны для просмотра как PDF-превью, при скачивании сохраняется исходный документ.
- Поддерживаются импорт ранее подписанных договоров, дополнительные соглашения с привязкой к основному договору, email-ссылки на конкретную задачу и админское версионирование шаблонов `.docx`.
- В карточке договора есть мини-чат: участники могут переписываться, прикладывать файлы и упоминать коллег через `@` без перезапуска маршрута; после финального статуса чат доступен только для чтения.
- Для доходного договора без ПСР поддерживается генерация `.docx`: ФНС заполняет реквизиты контрагента, юридический адрес добирается из PDF-выписки ФНС при отсутствии в быстром ответе, а подписи заказчика формируются с должностью, организацией, строкой подписи и `м.п.`.
- Для подписанта разделены формы ФИО и должности: родительный падеж используется в тексте договора, обычная форма должности — в блоке подписи. Банковские реквизиты проверяются до отправки формы.

### Бизнес-процесс проверки кандидатов
- Раздел `/business-processes/candidate-checks` доступен HR, руководителю СБ и администратору.
- HR (`head_hr`, `hr_specialist`) создаёт проверку кандидата и обязательно прикрепляет анкету.
- Руководитель СБ (`security`) скачивает анкету и фиксирует результат: `Согласован`, `Согласован с замечаниями` или `Не согласован`.
- Для результата с замечаниями или отказа комментарий обязателен.
- Администратор имеет полный доступ к БП: просмотр, создание проверки и решение СБ.
- Анкеты хранятся отдельно от договорных файлов в `uploads/candidate-checks/{candidateCheckId}`.
- Email-уведомления используют существующую SMTP-настройку: СБ получает письмо о новой проверке, HR получает письмо о результате. Анкета в письмо не вкладывается; письмо содержит ссылку на карточку проверки.

### Склад ТС
- Разделы `/warehouse`, `/warehouse/operations`, `/warehouse/reception`, `/warehouse/issue`.
- Кладовщик принимает ТС, выполняет фотофиксацию, добавляет дополнительные услуги и оформляет выдачу.
- Руководитель склада контролирует складские операции и может выполнять контролируемую корректировку дат с обязательной причиной.
- Финансист управляет тарифами, проверяет начисления и формирует акты/реестры.
- Представитель контрагента видит только свои ТС, услуги, начисления и акты; создавать или менять складские операции он не может.
- Excel-выгрузка начислений за выбранный период содержит два листа: `Акт` по хранению/приёмке/выдаче и `Доп услуги` по ручным услугам.
- Тарифы склада вводятся как суммы с НДС; ставка берётся из настроек НДС, а сумма НДС выделяется из тарифа при расчёте акта.
- Реестры ТС, клиентов склада и тарифов приведены к единому табличному виду; карточки открываются двойным кликом по строке, а служебные статусы отображаются текстом без лишних плашек.
- Фото приёмки и выдачи загружаются через устойчивый TUS-подход и хранятся в постоянных каталогах `/data`.

## 🧮 Паспорт формул (кратко)

Подробная версия: `ARCHITECTURE.md` → раздел **Паспорт формул (Planning v2)**.

- `planToDate = (planMonth / daysInMonth) * completedDays`
- Для прошлых месяцев: `planToDate = planMonth`
- Для будущих месяцев: `planToDate = 0`
- KTK: `Итого(план) = выгрузка/погрузка(план) + перемещение(план)`, аналогично для факта.
- AUTO waiting: `waiting[d] = waiting[d-1] + received[d] - sent[d]` (на 1-й день с `waitingStart`).
- RAIL totals: `из ВВО(итого) = 20 + 40`, `во ВВО(итого) = 20 + 40`, `жд(итого) = из + во`.
- RAIL received: `принято всего` — ввод по дням (сумма за месяц).
- RAIL waiting: `waiting[d] = waiting[d-1] + received[d] - sent[d]` (на 1-й день с `waitingStart`), где `sent = жд(итого)`, `received = принято всего`.
- AUTO debts: добавлены `Подрядчики Владивосток (₽)` (тип `currency`, агрегация `LAST`).
- EXTRA total: сумма 4 подсегментов.
- Carry-over (классический):
  - `carry(янв) = base(янв)`
  - `carry(m) = base(m) + debt(m-1)`
  - `debt(m) = max(0, carry(m) - fact(m))`
  - перевыполнение не создает "кредит" на следующий месяц.

- **Контейнеризация:** Docker + Docker Compose

- **Хостинг:** managed‑платформа (Amvera) или VPS
- **Резервное копирование:** ежедневные бэкапы
- **SSL сертификат:** Let's Encrypt

## ⚙️ Установка и запуск

### Предварительные требования
- Node.js 18+ и npm
- Docker и Docker Compose
- Git

### 1. Клонирование репозитория
```bash
git clone <repository-url>
cd Report
```

### 2. Запуск базы данных и Redis
```bash
cd backend
docker-compose up -d
```

Проверьте, что контейнеры запущены:
```bash
docker-compose ps
```

### 3. Настройка переменных окружения
Создайте файл `.env` в папке `backend` на основе `.env.example`:
```bash
cp .env.example .env
```

Отредактируйте `.env` при необходимости (порты, пароли, SMTP).

### 4. Установка зависимостей backend
```bash
cd backend
npm install
```

### 5. Инициализация базы данных
```bash
# Создание таблиц (TypeORM synchronize уже включен)
# При необходимости выполните SQL скрипты вручную:
docker exec -i logistics_postgres psql -U postgres -d logistics_reporting -f /docker-entrypoint-initdb.d/01-init.sql
```

### 6. Запуск backend в режиме разработки
```bash
npm run dev
```

Backend будет доступен по адресу: `http://localhost:3001`

### 7. Установка зависимостей frontend
```bash
cd ../frontend
npm install
```

### 8. Запуск frontend
```bash
npm run dev
```

Frontend будет доступен по адресу: `http://localhost:5173`

### 9. Вход в систему
Откройте `http://localhost:5173` в браузере и войдите под учетной записью администратора.

Рекомендуемый вариант:
- создать/задать админа через переменные окружения backend:
  - `DEFAULT_ADMIN_EMAIL`
  - `DEFAULT_ADMIN_PASSWORD`

## 📊 Использование

### Панель управления (Dashboard)
После входа открывается главная панель с ключевыми метриками по всем направлениям:
- Выполнение плана за текущий месяц
- Графики динамики показателей
- Уведомления о критических отклонениях

### Работа с направлением
1. Выберите направление в боковом меню.
2. Откроется таблица с оперативными данными (аналогичная Excel).
3. Вводите плановые и фактические значения – система автоматически рассчитает итоги и проценты выполнения.
4. Используйте кнопки «Сохранить», «Экспорт в Excel», «Сформировать отчет».

### Формирование отчетов
- **Ежедневный отчет:** доступен на странице направления после ввода данных.
- **Операционный отчет:** в меню «Показатели» → «Операционный отчет».
- **Сводный отчет:** в меню «Отчетность» → «Сводный отчет».
- Настройте периодичность email‑рассылки отчетов в разделе «Настройки» → «Email рассылка».
- Тип рассылки **«СВ за месяц (итоговый)»** отправляет отчёт за прошлый месяц.

### Согласование договоров
1. Инициатор, включая менеджера или руководителя отдела продаж, создает договор в разделе `Мои договоры`, добавляет документы и запускает согласование.
2. Для расходного договора и доходного договора с ПСР руководитель СБ, юрист, главный бухгалтер и финансовый директор получают задачи параллельно.
3. Для доходного договора без ПСР используется маршрут руководитель СБ → офис-менеджер.
4. После завершения обязательных виз офис-менеджер получает задачу `Передача на подпись`.
5. Офис-менеджер нажимает `Распечатать договор`: формируется единый PDF-пакет с листом согласования, вложениями согласующих и исходными документами договора.
6. После физической подписи офис-менеджер выбирает результат `Подписан` или `Не согласован`; для отказа нужен комментарий. Если скан получил инициатор, он может приложить его самостоятельно.

В реестре подписанный договор доступен пользователям, имеющим доступ к модулю БП. Незавершенные договоры доступны только инициатору, администратору и назначенным участникам маршрута.

### SW Tech Dashboard
- Для ролей `admin`, `director`, `financer`, `head_sales` стартовая страница — **SW Tech Dashboard** (`/sw-tech-dashboard`).
- Период выбирается в верхней синей панели: `Год` и `Месяц`.
- Данные дашборда строятся в режиме **«Итог месяца»**:
  - для выбранного периода используется `asOfDate = последний день выбранного месяца`;
  - это единое поведение для прошлого, текущего и будущего месяцев.
- График **«Динамика года: План / Факт / Выполнение %»**:
  - `План` = базовый месячный план (`basePlan`);
  - `% Выполнения` считается относительно базового плана.
- График направлений привязан к выбранному месяцу (`month_segments`, legacy-алиас: `april_segments`).
- Карточки поддерживают drill-down:
  - карточки направлений → `Ежедневный отчет` (`/plans?segment=...&year=...&month=...`);
  - карточка `Итого за <месяц>` → `Операционный отчет` (`/plans/totals?year=...`).

### График работы (Operations Preview)
- Подменю **«Показатели»** и **«График работы»** работают как ручные `toggle`.
- Раскрытие/сворачивание не меняется автоматически при переходе между разделами.
- Состояние раскрытий сохраняется в `localStorage` и восстанавливается после перезагрузки.
- Для Владивостока доступны отдельные графики `Диспетчерский отдел`, `Гараж` и
  `Служба Безопасности -> Сотрудник охраны`; график службы безопасности использует отдельное хранилище
  `security_preview_v1`, поддерживает `План/Факт` и статус `Н — нет сотрудника`.
- `Руководитель СБ` (`security`) ведет только график сторожей; администратор и роли
  отдела кадров могут редактировать график и выгружать его в отчетах.
- Тестовая кнопка автозаполнения `ТЕСТ 17` удалена; тестовые записи формата `test-*` / `Тест Водитель ...` очищаются при восстановлении состояния.

### Валовая прибыль, план
- Для строки «Цена с НДС, руб/шт» в колонке «Итог за год» используется **среднее арифметическое** по месяцам, а не сумма.

### Администрирование
- Раздел «Администрирование» доступен только пользователям с ролью `admin`.
- Возможности: управление пользователями, системная статистика, журнал действий.

### Настройки (admin)
- «Уведомления»: расписания email‑рассылок.
- «Почта»: параметры SMTP.
- «НДС»: управление ставками по годам и периодам.

### Примечание для Amvera
- Текущий Amvera-деплой использует один Docker-контейнер: backend раздаёт API,
  WebSocket, scheduler и собранный frontend.
- Redis обязателен только при `SCHEDULER_USE_QUEUE=true`. Для stage можно
  использовать `SCHEDULER_USE_QUEUE=false`, тогда включается локальный fallback
  планировщика.
- Для деплоя используйте `amvera.yml` в корне репозитория.
- Для постоянных файлов на Amvera используйте `/data`:
  - `UPLOAD_PATH=/data/uploads`
  - `WAREHOUSE_UPLOAD_PATH=/data/warehouse`
  - `WAREHOUSE_PHOTO_BACKUP_PATH=/data/warehouse-photo-backup`
  - `WAREHOUSE_TUS_TEMP_PATH=/data/warehouse-tus-temp`
- Перед релизом используйте чеклист: `backend/PRODUCTION_CHECKLIST.md`.
- Для корректного обновления фронтенда используются cache‑заголовки:
  - `index.html` → `Cache-Control: no-cache`
  - `/assets/*` → `Cache-Control: public, max-age=31536000, immutable`

## 🛡️ Надежность

### Health-checks
- `GET /health` — жив ли процесс.
- `GET /health/db` — проверка доступности БД (`SELECT 1`, таймаут 2s, 2 попытки).
- `GET /health/db/metrics` — метрики по доступности БД (latency + ошибки).
- `GET /health/redis` — проверка Redis для очередей/планировщика; возвращает `NOT_REQUIRED`, если очередь отключена.
- `GET /health/scheduler` — статус планировщика email.

### Circuit breaker для БД
Если БД недоступна, API временно возвращает `503`, чтобы не создавать лавину запросов.

Настройки через переменные окружения:
- `DB_CIRCUIT_FAILURE_THRESHOLD` (по умолчанию `5`)
- `DB_CIRCUIT_OPEN_MS` (по умолчанию `30000`)
