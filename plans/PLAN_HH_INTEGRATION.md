# План интеграции hh.ru — HR-модуль (поиск кандидатов и скачивание резюме)

Проект: **Report** (backend: Node.js + TypeScript, Express, TypeORM/PostgreSQL; frontend: React + TS; деплой: Amvera)
Дата: 21.07.2026
Статус: план, к реализации не приступали

---

## Цель

HR-специалисты (`head_hr`, `hr_specialist`) работают внутри модуля под своими учётными записями hh.ru:
ищут кандидатов по базе резюме, открывают резюме, скачивают их в PDF/RTF, приглашают на вакансии —
без ручного переноса данных из hh.

Ключевой принцип архитектуры: **модуль — витрина поверх hh, а не хранилище резюме**.
Данные резюме подтягиваются по запросу и не складываются в собственную базу
(требование «Условий использования сервиса API» hh и 152-ФЗ).
Модуль анкеты трудоустройства (`candidate_checks`) — отдельный внутренний БП и с hh не пересекается.

## Этап 1. Конфигурация и хранение токенов

### 1.1 Переменные окружения — `backend/src/config/env.ts`

Добавить по образцу существующих геттеров (`getJwtSecret` и т.д.):

```
HH_CLIENT_ID
HH_CLIENT_SECRET
HH_REDIRECT_URI
HH_USER_AGENT          # формат: "ReportApp/1.0 (hr@<домен компании>)" — обязательное требование hh
HH_TOKEN_ENC_KEY       # ключ шифрования токенов в БД (32 байта, base64)
```

Для `HH_CLIENT_SECRET` и `HH_TOKEN_ENC_KEY` — то же правило, что для `JWT_SECRET`:
в production отсутствие значения = ошибка старта (`assertProductionEnv`). Без fallback-значений.
На Amvera завести секреты в переменных окружения проекта.

### 1.2 Модель и миграция — токены на пользователя

Новые файлы:

- `backend/src/models/hh-account.model.ts` — таблица `hh_accounts`:
  - `id` (uuid), `user_id` (FK → users, unique — одна привязка на сотрудника)
  - `hh_user_id`, `hh_email` (для отображения «подключено как …»)
  - `access_token_enc`, `refresh_token_enc` — **шифрованные** (AES-256-GCM, ключ из env)
  - `expires_at`, `scopes`, `created_at`, `updated_at`
- `backend/src/migrations/<ts>-AddHhAccounts.ts` — по образцу `1783120000000-AddCandidateChecks.ts`

Пароли hh нигде не появляются — только OAuth-токены (прямое требование hh).

### 1.3 Учёт квот hh

- `backend/src/models/hh-usage.model.ts` — таблица `hh_usage_daily`:
  `user_id`, `date`, `resume_views`, `searches` — счётчик на менеджера в день.
  Нужна, чтобы показывать HR остаток дневного лимита просмотров и мягко останавливать до `429`.

## Этап 2. OAuth-подключение учётки hh

Новые файлы: `backend/src/routes/hh.routes.ts`, `backend/src/controllers/hh.controller.ts`,
`backend/src/services/hh-auth.service.ts`.

Маршруты (все под существующим `authenticate` + `authorize(['head_hr','hr_specialist','admin'])`):

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/hh/connection` | статус привязки текущего пользователя («подключено как …») |
| GET | `/api/hh/authorize` | выдать URL `https://hh.ru/oauth/authorize?response_type=code&client_id=...&state=<csrf>` |
| GET | `/api/hh/callback` | обмен `code` → `access_token`/`refresh_token` (POST `https://api.hh.ru/token`), проверка `state`, сохранение в `hh_accounts`, редирект во фронт |
| DELETE | `/api/hh/connection` | отвязать учётку (удалить токены) |

`hh-auth.service.ts`:

- `exchangeCode(code)`, `refreshTokens(userId)` — refresh за ~5 мин до `expires_at`, с одиночным «полётом» (mutex), чтобы параллельные запросы не сжигали refresh_token;
- шифрование/дешифрование токенов;
- при невалидном refresh — пометить привязку разорванной и вернуть фронту сигнал «переподключите hh».

CSRF `state` хранить короткоживуще (по образцу `utils/cache.ts`).

## Этап 3. Клиент hh API

Новый файл: `backend/src/services/hh-client.service.ts` — единая точка всех запросов к `api.hh.ru`
(паттерн уже отработан в `fns-egrul.service.ts`: fetch + заголовки + цепочка «запрос → статус → скачивание»).

Обязанности клиента:

- заголовки на каждый запрос: `Authorization: Bearer <token>`, `HH-User-Agent`/`User-Agent` из `HH_USER_AGENT` (не `Mozilla/5.0`);
- `401` → одна попытка refresh → повтор запроса → иначе «переподключите hh»;
- `403` (контакты недоступны — нет оплаченных услуг) → типизированная ошибка `HhContactsUnavailableError`;
- `429` (дневной лимит) → типизированная ошибка `HhLimitExceededError` + фиксация в `hh_usage_daily`;
- `503`/сеть → retry с backoff (по образцу `utils/db-retry.ts`), затем аналог `FnsServiceUnavailableError`;
- инкремент счётчиков квот при каждом просмотре резюме.

Методы:

```
searchResumes(userId, params)      → GET  /resumes?text=&area=&salary_from=&experience=&page=&per_page=
getResume(userId, resumeId)        → GET  /resumes/{id}          (+1 к счётчику просмотров)
downloadResume(userId, url, fmt)   → GET  <download.pdf.url | download.rtf.url> — стрим файла
getVacancies(userId)               → GET  /vacancies?employer_id= (для приглашений)
inviteToVacancy(userId, ...)       → POST /negotiations           (опционально, этап 6)
```

Скачивание: URL берётся **только** из поля `download` объекта резюме (ссылки динамические,
привязаны к токену), файл проксируется пользователю стримом через существующий
`utils/content-disposition.ts`. На диск сервера не сохраняется.

## Этап 4. Backend-маршруты модуля

`backend/src/routes/hh.routes.ts` (продолжение), валидация через `express-validator`
по образцу `candidate-checks.routes.ts`:

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/api/hh/resumes` | поиск по базе (проксирование параметров, пагинация hh) |
| GET | `/api/hh/resumes/:id` | карточка резюме (транзитом, без записи в БД) |
| GET | `/api/hh/resumes/:id/download?format=pdf\|rtf` | стрим файла резюме |
| GET | `/api/hh/dictionaries` | справочники hh (регионы, опыт) с кэшем в `utils/cache.ts` (сутки) |
| GET | `/api/hh/usage` | остаток дневных лимитов текущего HR |

Аудит: писать факт просмотра/скачивания в существующий `audit-log.service.ts`
(кто, когда, `resume_id` — без содержимого резюме).

### Политика хранения (фиксируется в коде и в доке)

**Храним:** `resume_id`, url резюме на hh, дату/факт просмотра, счётчики квот.
**Не храним:** ФИО, контакты, текст резюме, файлы PDF/RTF из базы hh.
Если кандидат из hh идёт в найм — он заполняет вашу анкету (`candidate_checks`) сам, как и сейчас;
автопереноса данных из hh в анкету на этом этапе не делаем.

По существующей структуре `frontend/src/`:

## Порядок работ и оценка

```
0 (регистрация) ──────────────► 5 (модерация — ждём hh)
1 (конфиг+БД) → 2 (OAuth) → 3 (клиент) → 4 (роуты) → 5 (фронт) → 7 (тесты/деплой)
```

Грубая оценка чистой разработки: этапы 1–2 ≈ 3–4 дня, этап 3 ≈ 2–3 дня, этап 4 ≈ 2 дня,
этап 5 ≈ 4–5 дней, этап 7 ≈ 2–3 дня. Итого ~2.5–3 недели одним разработчиком,
плюс внешние сроки модерации hh.
