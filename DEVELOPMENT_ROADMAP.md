# План дальнейшей разработки (Roadmap)

Документ фиксирует следующий этап развития проекта после текущей версии в продакшене.

## 1) Цели следующего этапа

- Довести расчеты и отчеты до полного совпадения с бизнес-логикой и Excel-эталоном.
- Повысить надежность продакшена (Redis, scheduler, ошибки 5xx, наблюдаемость).
- Закрыть UX-доработки для ежедневной работы менеджеров.
- Подготовить процесс безопасных релизов и поддержки.
- Показатели эффективности — дальнейшая разработка.

## 2) Приоритеты (по порядку)

1. **Критическая стабильность продакшена**
2. **Корректность расчетов и отчетов**
3. **Регламент и удобство ежедневной эксплуатации**
4. **Техдолг, тесты и масштабирование**

## 2.1) Текущая разработка

### Ежемесячный обзор по водителям и ТС
- Единая таблица по дням месяца для водителей и ТС.
- Разделение по типам ТС (контейнеровозы / автовозы).
- Поддержка 1–2 водителей на одну машину, отдельные строки и примечания.
- Быстрый ввод статусов по дням, наглядность как в Excel‑графике.

## Этап A — Стабилизация продакшена (1–3 дня)

### A1. Redis и scheduler
- Проверить соединение с Redis в Amvera (host/port/password, таймауты, retries).
- Добавить явный health-check Redis и понятный статус scheduler в логах/эндпоинтах.
- Убедиться, что при недоступности Redis backend не падает, а корректно деградирует.
- Добавить ретраи отправки email при временных сбоях (backoff, не ронять весь цикл).

**Критерий приемки**
- Нет падений приложения из-за Redis.
- Рассылки по расписанию стабильно выполняются.
- В логах есть явные сообщения `scheduler started/stopped/error`.

### A2. Ошибки 502/5xx
- Локализовать проблемные запросы (особенно массовые сохранения и редактирование планов).
- Добавить ограничение частоты/дебаунс на клиенте там, где создаются “серии” запросов.
- Добавить защиту от race conditions в API сохранения.

**Критерий приемки**
- При обычной работе в UI нет 502.
- Серии быстрых правок не приводят к неконсистентным данным.

## Этап B — Полная корректность бизнес-логики (2–4 дня)

### B1. План с переносом (единая логика)
- Применить и проверить логику во всех сегментах:
  - `carry_m = base_m + debt_prev`
  - `debt_prev = max(0, carry_m - fact_m)`
  - перевыполнение не дает “кредит в минус”.
- Для строки “Итог за год / План с переносом” оставить значение как у базового годового плана (по согласованному правилу отображения).

**Критерий приемки**
- Совпадение с тест-кейсами по ТО/КТК/Авто/ЖД.
- Дашборды и Операционный отчет не расходятся по `planMonth`.

### B2. AUTO (ожидание)
- Зафиксировать правило:
  - дневные ячейки ожидания — накопительные,
  - колонка “ИТОГО” для ожидания — **последнее значение месяца**, а не сумма.
- Проверить перенос остатков между месяцами и на границе года (декабрь -> январь).

**Критерий приемки**
- Для AUTO значения “В ожидании” в ИТОГО всегда соответствуют последнему дню периода.

### B3. Сводный отчет
- Проверить агрегирование подсегментов:
  - Автовозы/Шторы объединенно,
  - Авто в КТК отдельно,
  - ЖД объединенно “Из/Во Владивосток” согласно согласованной схеме.

**Критерий приемки**
- Сводный отчет совпадает с Операционным отчетом и дашбордами по тем же периодам.

## Этап C — UX и операционная удобность (2–3 дня)

### C1. Ежедневный отчет
- Финальный визуал дашбордов (единый размер карточек, выравнивание).
- Явная подсветка вычисляемых строк + блокировка редактирования.

### C2. Предупреждения о несохраненных данных

### C3. Email-шаблоны
- Унифицировать темы и тело писем для:
  - отчетов по сегментам,
  - Сводного отчета.
- Проверить русскую локализацию и формат дат `dd.mm.yyyy`.

**Критерий приемки**
- Бизнес-пользователь получает понятные письма с корректными вложениями и темами.

## Этап D — Качество, поддержка и релизы (2–5 дней)

### D1. Тесты
- Расширить unit-тесты:
  - перенос плана (в т.ч. перевыполнение),
  - AUTO waiting,
  - сводные агрегаты.
- Добавить smoke-checklist для ручной регрессии.

### D2. Техдолг
- Удалить/архивировать устаревшие части legacy-функционала (где подтверждено, что не используются).
- Привести названия ролей/экранов/метрик к единому словарю.

### D3. Релизный процесс
- Релиз через `main` + короткий changelog.
- Мини-чеклист перед каждым прод-деплоем:
  - миграции,
  - переменные окружения,
  - здоровье Redis/DB,
  - тестовая рассылка.

**Критерий приемки**
- Прогнозируемый деплой без “сюрпризов”.
- Понятная процедура отката.

## 3) Риски и меры

- **Риск:** рассинхрон между Excel-логикой и реализацией.  
  **Мера:** фиксированный набор эталонных тест-кейсов (янв/фев/март).

- **Риск:** Redis-сбои в managed-среде.  
  **Мера:** graceful degradation + мониторинг + отдельный health-check.

- **Риск:** “тихие” ошибки при массовом редактировании.  
  **Мера:** идемпотентные API, дебаунс, логирование входных батчей.

## 6) Аудит безопасности и надежности (предложения)

Ниже — результаты аудита с рекомендациями по ключевым модулям. Формат: **что важно → риски → рекомендации**.

### 6.1 Авторизация и пользователи (Auth + Users)
**Ключевые файлы:**  
`backend/src/controllers/auth.controller.ts`  
`backend/src/routes/auth.routes.ts`  
`backend/src/middleware/authenticate.ts`  
`frontend/src/store/auth-store.ts`  
`frontend/src/services/api.ts`

**Что важно**  
Это ворота системы. Любая ошибка здесь — риск полного доступа к данным.

**Риски**
- Токен хранится в `localStorage` (`frontend/src/store/auth-store.ts`), уязвимость к XSS.
- JWT отдается в ответе (`backend/src/controllers/auth.controller.ts`) и используется только в `Authorization` header, нет refresh‑логики.
- Password reset делается через JWT, но нет дополнительной ротации сессий.
- В `auth.routes.ts` есть rate‑limit, но storage in‑memory и не работает между инстансами.

**Рекомендации**
- Перейти на httpOnly cookie + CSRF, либо хотя бы CSP + убрать `localStorage` для токена.
- Добавить refresh‑token или короткие access‑token и стратегию обновления.
- Использовать Redis‑лимитер.
- Ужесточить логирование и аудит входов/выходов.

### 6.2 Админка (Users/Admin)
**Ключевые файлы:**  
`backend/src/controllers/admin.controller.ts`  
`backend/src/routes/admin.routes.ts`

**Что важно**  
Админ — самая опасная точка. Здесь управляются роли, пароли, пользователи.

**Риски**
- Сброс пароля возвращает временный пароль в ответе (`resetUserPassword`).
- Удаление пользователя сопровождается raw SQL `UPDATE` без доп. проверки.
- Все админские операции идут через JWT без второго фактора.

**Рекомендации**
- Не возвращать временный пароль в API (только email/ссылка).
- Логировать все админ‑действия, включая изменение ролей.
- Добавить MFA или хотя бы подтверждение критических операций.

### 6.3 Email‑рассылки (Schedulers + SMTP)
**Ключевые файлы:**  
`backend/src/services/email-scheduler.service.ts`  
`backend/src/controllers/email-schedule.controller.ts`  
`backend/src/controllers/smtp-config.controller.ts`  
`frontend/src/pages/SettingsPage.tsx`

**Что важно**  
Это операционная часть. Ошибки здесь = неверные отчеты или спам.

**Риски**
- SMTP‑пароль хранится в открытом виде и отдается фронту.
- Валидация расписаний слабая — можно записать невалидные `department/reportType/timezone`.
- Локальный scheduler fallback может запускаться параллельно при нескольких инстансах → дубль отправок.
- Нет явного контроля “что входит в отправку” (можно ошибочно задублировать отчеты).

**Рекомендации**
- Хранить SMTP‑пароли только в backend (шифрование + не выдавать на frontend).
- Добавить строгую валидацию схем расписаний (Joi/Zod).
- Использовать Redis/Bull как единую точку расписания.
- Добавить защиту от двойных отправок (lock + `lastSent`).

### 6.4 Планирование v2 (Planning)
**Ключевые файлы:**  
`backend/src/controllers/planning-v2.controller.ts`  
`backend/src/services/planning-v2.service.ts`  
`backend/src/services/planning-v2-report.service.ts`

**Что важно**  
Ключевая бизнес‑логика. Ошибки = неверные цифры в отчетах.

**Риски**
- Проверка прав основана только на `role` без привязки к конкретному сегменту в БД.
- Потенциальные ошибки в данных при параллельных апдейтах (`batchUpsert`).
- Логика доступа разнесена между сервисом и контроллером → риск несогласованности.

**Рекомендации**
- Привязать права к данным пользователя (например, `user.segmentAccess`).
- Защитить `batchUpsert` транзакцией с ясной моделью конфликтов.
- Добавить тесты на границы ролей и сегментов.

### 6.5 Финансовый план
**Ключевые файлы:**  
`backend/src/services/financial-plan.service.ts`  
`backend/src/controllers/financial-plan.controller.ts`  
`frontend/src/pages/SettingsPage.tsx`

**Что важно**  
Высокая цена ошибки → финансовые решения.

**Риски**
- Есть операции “вручную” без строгой историзации.
- Нет лимитов на размер вводимых данных.
- Экспорт/импорт Excel без строгой проверки корректности.

**Рекомендации**
- Добавить аудит изменений значений (кто/когда/что).
- Жесткие проверки диапазонов.
- Утвердить формат Excel (валидация структуры).

### 6.6 Заметки и календарь
**Ключевые файлы:**  
`backend/src/controllers/notes.controller.ts`  
`backend/src/models/note*.ts`  
`frontend/src/pages/CalendarPagePreview.tsx`

**Что важно**  
Организация коммуникаций. Ошибки = утечка частных заметок.

**Риски**
- Доступ к заметкам проверяется по роли/автору, но нет ограничений по времени доступа.
- Нечеткое разделение “личные/для всех/адресные”.
- Автор и получатели показываются всем, а не только адресатам (в UI).

**Рекомендации**
- Уточнить бизнес‑правила: кто должен видеть автора и получателей.
- Добавить флаг видимости для полей, а не только заметки целиком.
- Тесты на доступы (админ, автор, адресат, неадресат).

### 6.7 WebSocket / realtime
**Ключевые файлы:**  
`backend/src/services/websocket.service.ts`

**Что важно**  
Потоковые данные без auth = риск утечки.

**Риски**
- Нет авторизации на WebSocket.
- Нет фильтрации событий по ролям.

**Рекомендации**
- Ввести токен в параметрах соединения.
- Фильтровать события по доступным сегментам/ролям.

### 6.8 Импорт/экспорт Excel
**Ключевые файлы:**  
`backend/src/services/excel-import-export.service.ts`

**Что важно**  
Импорт плана = основа финансового блока.

**Риски**
- `importPlansFromExcel` не дожидается `savePlan` (асинхронные вызовы).
- Возможна частичная запись без пересчёта.

**Рекомендации**
- Собирать промисы и `await Promise.all`.
- Пересчитывать планы только после успешного импорта.

### 6.9 Конфигурация и инфраструктура
**Ключевые файлы:**  
`backend/src/config/data-source.ts`  
`backend/.env`  
`backend/src/app.ts`

**Что важно**  
База и безопасность.

**Риски**
- `synchronize: true` всегда.
- Дефолтные креды `postgres/postgres`.
- Настройки CORS могут выбрасывать ошибку.

**Рекомендации**
- Отключить `synchronize` в проде.
- Обязательные env‑проверки при старте.
- Вернуть CORS error как 403, а не 500.
