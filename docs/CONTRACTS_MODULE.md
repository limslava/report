# Модуль «Согласование договоров»

Документ фиксирует текущее состояние модуля договоров, его роли, ограничения,
файлы, маршруты и особенности развёртывания. Используется как ориентир перед
деплоем stage/production и при проверке регрессий после изменений в других
модулях.

## Назначение

Модуль автоматизирует бизнес-процесс согласования договоров:

- создание черновика договора инициатором;
- прикрепление файлов договора и дополнительных документов;
- запуск маршрута согласования;
- сбор решений согласующих;
- формирование печатного PDF-пакета;
- фиксация подписанного экземпляра;
- просмотр подписанных договоров в реестре.

## Основные маршруты

- UI:
  - `Мои договоры`;
  - `Согласование договоров`;
  - `Реестр договоров`;
  - inbox согласующих;
  - карточка договора / лист согласования.
- API:
  - `/api/contracts/*`;
  - `/api/contracts/:id/start-approval`;
  - `/api/contracts/:id/approval-sheet`;
  - `/api/contracts/:id/print-package`;
  - `/api/contracts/:id/attachments`;
  - `/api/contracts/attachments/:attachmentId/download`;
  - `/api/contracts/attachments/:attachmentId/preview`.

## Роли и доступ

### Инициаторы

Инициаторами могут быть:

- `manager_sales`;
- `head_sales`;
- `admin`.

Инициатор может:

- создать черновик договора;
- редактировать свой черновик;
- прикреплять файлы договора;
- запускать договор на согласование;
- подготовить новую редакцию;
- удалить свой черновик до запуска;
- прикрепить подписанный экземпляр как резервный сценарий, если скан получил
  инициатор.

Инициатор не получает автоматически права согласующего. Поставить визу может
только назначенный участник маршрута или администратор.

### Согласующие

В маршруте используются роли:

- `security` / руководитель СБ — первичная проверка;
- `lawyer` — юридическое согласование;
- `chief_accountant` — бухгалтерское согласование;
- `financer` — финансовое согласование;
- `secretary` / офис-менеджер — передача на физическую подпись.

После решения руководителя СБ договор одновременно поступает юристу, главному
бухгалтеру и финансовому директору. Решение одного параллельного согласующего
не отменяет решения остальных.

### Администратор

`admin` имеет расширенный доступ:

- просмотр всех договоров;
- контроль маршрута;
- просмотр истории решений;
- управляемое исправление данных;
- удаление вложений при необходимости.

## Маршрут согласования

1. Инициатор создаёт черновик и прикладывает документы.
2. Инициатор запускает согласование.
3. Руководитель СБ выполняет первичную проверку.
4. После СБ параллельно назначаются:
   - юрист;
   - главный бухгалтер;
   - финансовый директор.
5. После завершения параллельного блока договор передаётся офис-менеджеру.
6. Офис-менеджер формирует печатный пакет и передаёт договор на физическую
   подпись.
7. После подписания офис-менеджер прикрепляет скан подписанного экземпляра.
8. Если подписанный экземпляр получил инициатор, он может прикрепить его сам.
9. Подписанный договор становится доступен в реестре пользователям, которым
   открыт модуль согласования договоров.

Электронная виза генерального директора в маршруте не создаётся. Генеральный
директор указывается в печатном листе согласования как лицо, подписывающее
документ на бумаге.

## Решения согласующих

Согласующий может:

- согласовать;
- согласовать с замечаниями;
- не согласовать;
- оставить комментарий;
- прикрепить документы к своей визе.

Комментарий обязателен при решении «Согласован с замечаниями».

Вложения согласующих включаются в печатный пакет и доступны участникам процесса.

## Файлы и PDF

Файлы договора хранятся в постоянном каталоге:

```env
UPLOAD_PATH=/data/uploads
```

Для Amvera этот путь должен находиться на persistence mount `/data`. Нельзя
использовать временный каталог контейнера как постоянное хранилище договорных
документов.

DOCX-файлы:

- не заменяются при предпросмотре;
- сохраняются в исходном виде;
- доступны для скачивания как оригинал;
- преобразуются в PDF только для предпросмотра и печатного пакета.

Для server-side DOCX-to-PDF preview в контейнере нужен LibreOffice:

```env
LIBREOFFICE_BIN=libreoffice
```

Dockerfile уже устанавливает `libreoffice-writer`, `fonts-dejavu-core` и
`fonts-liberation`.

## Печатный пакет

Печатный пакет формируется одним PDF в порядке:

1. лист согласования;
2. вложения согласующих;
3. исходные документы договора.

Пакет используется офис-менеджером для передачи на физическую подпись.

## Видимость документов

- Незавершённый договор доступен:
  - администратору;
  - инициатору;
  - назначенным участникам маршрута.
- Подписанный договор доступен пользователям, которым открыт модуль БП.
- Удаление вложений ограничивается ролью и владельцем документа.
- Удаление файлов договора администратором считается контролируемой операцией
  исправления данных.

## Уведомления и scheduler

Модуль использует email/scheduler для сервисных уведомлений и обработки
регламентных задач.

Для stage можно использовать локальный scheduler без Redis:

```env
SCHEDULER_ENABLED=true
SCHEDULER_USE_QUEUE=false
SCHEDULER_INTERVAL_MINUTES=5
SCHEDULER_TIMEZONE=Asia/Vladivostok
REDIS_ENABLED=false
```

Если используется Redis-очередь:

```env
SCHEDULER_USE_QUEUE=true
REDIS_ENABLED=true
REDIS_HOST=<redis-host>
REDIS_PORT=6379
REDIS_PASSWORD=<redis-password>
```

SMTP нужен для фактической отправки писем:

```env
SMTP_HOST=<smtp-host>
SMTP_PORT=587
SMTP_SECURE=false
SMTP_USER=<smtp-user>
SMTP_PASSWORD=<smtp-password>
SMTP_FROM=<from-email>
```

## База данных

Для production:

```env
DB_SYNCHRONIZE=false
DB_MIGRATIONS_RUN=true
```

Для stage после крупных изменений схемы временно может использоваться
`DB_SYNCHRONIZE=true`, но это временная мера до перевода изменений в TypeORM
migrations. Перед production-релизом схема должна обновляться миграциями.

Исторические SQL-миграции БП договоров перечислены в
`DEPLOY_WORKFLOW_AMVERA.md`.

## Post-deploy smoke test

Минимальная проверка после деплоя:

1. `/health` отвечает `200`.
2. `/health/db` отвечает `200`.
3. Логин работает.
4. Инициатор создаёт черновик договора.
5. Инициатор прикрепляет файл договора.
6. Инициатор запускает согласование.
7. Руководитель СБ видит договор в своём inbox.
8. Согласующий может открыть карточку, поставить решение и комментарий.
9. Офис-менеджер видит этап передачи на подпись.
10. Печатный пакет формируется.
11. DOCX открывается как PDF-превью.
12. «Скачать оригинал» отдаёт исходный DOCX.

## Связь со складским модулем

Модуль договоров и складской модуль используют общий справочник контрагентов,
но не должны смешивать бизнес-логику:

- договоры отвечают за согласование договорных документов;
- склад отвечает за хранение ТС, фотофиксацию, услуги, выдачу и начисления;
- складские клиенты ведутся отдельно через `warehouse_clients`;
- представитель контрагента склада привязывается к конкретному складскому
  клиенту и видит только свои ТС/начисления.
