## 2. CRITICAL

### C1. Захардкоженный админ со слабым паролем (3 места)
- **Категория:** Security · **Файлы:**
  - `backend/src/services/bootstrap.service.ts:8-9,16-25` (вызывается из `index.ts:7`)
  - `backend/.env.example:23-24` (в git)
  - `database/init/02-seed.sql:1-10` (пароль `admin123`, плейнтекст в комментарии рядом с хешем)
- **Суть:** при старте `ensureDefaultAdmin()` создаёт админа с fallback-значениями `2720233@gmail.com` / `312046`, если env-переменные не заданы. Те же значения закоммичены в `.env.example`; init-seed отдельно создаёт `admin@example.com` / `admin123`. Любой деплой, забывший переопределить переменные, стартует с публично известным супер-пользователем → полный захват системы. Значения уже в истории git.
```ts
const DEFAULT_ADMIN_EMAIL = process.env.DEFAULT_ADMIN_EMAIL || '2720233@gmail.com';
const DEFAULT_ADMIN_PASSWORD = process.env.DEFAULT_ADMIN_PASSWORD || '312046';
```
- **Рекомендация:** убрать fallback — падать при отсутствии `DEFAULT_ADMIN_*` (как уже сделано для `JWT_SECRET`); генерировать случайный пароль и требовать смену при первом входе; заменить реальные значения в `.env.example` на плейсхолдеры; не документировать плейнтекст в seed; ротировать пароль и почистить историю git.

### C2. Обход аутентификации при TUS-загрузке фото (роль из клиентских метаданных)
- **Категория:** Security · **Файл:** `warehouse-tus.service.ts:137-143, 200-216`
- **Суть:** после завершения загрузки пользователь определяется как `const user = resolvedUser ?? metadataUser`. Если валидного bearer-токена нет, берётся `metadataUser`, чьи `id`, `fullName` и **`role`** читаются напрямую из метаданных загрузки, контролируемых клиентом. Проверка `WAREHOUSE_TUS_ROLES.has(user.role)` тем самым доверяет роли, присланной клиентом → неаутентифицированная запись файлов под любой ролью.
```ts
const resolvedUser = await resolveTusUser(req);
const metadataUser = getMetadataUser(metadata); // role из metadata.uploadedByRole
const user = resolvedUser ?? metadataUser;
if (!user || !WAREHOUSE_TUS_ROLES.has(user.role)) { /* reject */ }
```
- **Рекомендация:** требовать и проверять JWT для каждой загрузки, полностью убрать fallback на `metadataUser`; метаданные использовать только для отображения после подтверждения роли из проверенного токена.

### C3. Path traversal при записи файлов фото
- **Категория:** Security · **Файлы:** `warehouse-photo-storage.service.ts:25-32,64-143`; питается из `warehouse-tus.service.ts:113-115,193`
- **Суть:** пути формируются как `path.join(storageRoot, vehicleId)` и `path.join(storageRoot, '_pending', uploadSessionId)` без валидации traversal (в отличие от read-пути `resolveStoredFile`, где есть проверка `startsWith`). `uploadSessionId`/`vehicleId` приходят из TUS-метаданных через `normalizeMetadataValue`, который только обрезает строку и не удаляет `/` и `..`. Значение вида `../../../var/www/...` выйдет за пределы каталога хранения. В связке с C2 — неаутентифицированная запись в произвольный каталог.
- **Рекомендация:** валидировать `vehicleId`/`uploadSessionId` как строгий UUID (или `[A-Za-z0-9_-]`), применять проверку `resolve(...).startsWith(root + sep)` ко всем write/rename/mkdir операциям.

### C4. Fallback JWT-секрет вне production
- **Категория:** Security · **Файл:** `backend/src/config/env.ts:3,17-23`
- **Суть:** при отсутствии `JWT_SECRET` используется литерал `'fallback-secret'`; отклоняется только при `NODE_ENV === 'production'`. Любой staging / окружение с несконфигурированным `NODE_ENV` подписывает и проверяет токены публично известным секретом → возможна подделка admin-JWT. (Dockerfile ставит `NODE_ENV=production`, поэтому прод в образе защищён — но не защищены прочие запуски.)
- **Рекомендация:** требовать `JWT_SECRET` безусловно (падать на старте при отсутствии), проверять минимальную длину/энтропию, никогда не поставлять константу.

### C5. Некорректный синтаксис миграций `CREATE TYPE IF NOT EXISTS`
- **Категория:** Bug · **Файлы:** `backend/database/migrations/20260430_create_contracts.sql:4-5`; `20260430_contract_approval_sheet_mvp.sql:1-5`
- **Суть:** PostgreSQL (вплоть до 16) не поддерживает `IF NOT EXISTS` у `CREATE TYPE` — эти операторы дают синтаксическую ошибку, чистый прогон миграций падает. При этом в TS-миграции `1783080000000` тот же приём сделан корректно через `DO $$ ... pg_type` — то есть правильный паттерн известен.
```sql
CREATE TYPE IF NOT EXISTS contract_type_enum AS ENUM ('expense', 'income');
```
- **Рекомендация:** заменить на блок `DO $$ BEGIN IF NOT EXISTS (SELECT 1 FROM pg_type WHERE typname='...') THEN CREATE TYPE ... END IF; END $$;`.

## 3. HIGH

### H1. Эскалация привилегий через self-registration
- **Security** · `auth.controller.ts:79-121`, `auth.routes.ts:45-57`
- `register` берёт `role` прямо из тела запроса (валидируется только `isIn(ROLE_VALUES)`, а туда входит `admin`). Если `INVITE_ONLY` когда-либо станет `false`, любой аноним зарегистрируется как admin. Плюс на `/register` нет rate-limit. **Фикс:** назначать роль по умолчанию с минимальными правами; выдача привилегированных ролей — только через аутентифицированный admin-эндпоинт; добавить лимитер.

### H2. Rate-limiter доверяет `X-Forwarded-For` (обход брутфорс-защиты логина)
- **Security** · `middleware/rate-limit.ts:15-21,40`
- `getClientKey` берёт сырой заголовок `X-Forwarded-For` до `req.ip`. Клиент управляет заголовком → на каждый запрос новый бакет, лимит логина (`max:10`) обходится. **Фикс:** использовать `req.ip` из `trust proxy`, не парсить заголовок вручную.

### H3. In-memory rate-limiter — по процессу и не персистентный
- **Bug** · `middleware/rate-limit.ts:24,44-71`
- Состояние в `Map` процесса: при нескольких инстансах лимит умножается на их число, при рестарте сбрасывается. **Фикс:** хранить лимиты в Redis (уже в зависимостях) / `rate-limiter-flexible`.

### H4. Пароль отправляется в почте открытым текстом
- **Security** · `auth.controller.ts:103`
- `sendInvitationEmail(email, fullName, role, password)` — плейнтекст-пароль по e-mail (логируется, долго хранится). **Фикс:** отправлять одноразовую короткоживущую ссылку на установку пароля.

### H5. SMTP-пароль возвращается в API и хранится плейнтекстом
- **Security** · `smtp-config.controller.ts:9-20`; модель `smtp-config.model.ts:20-21`
- `GET /api/smtp-config` отдаёт всю строку `res.json(config || {})` вместе с полем `password`; колонка — обычный `varchar` без `select:false`/`@Exclude`, без шифрования. Утечка учётки почтового сервера при любом admin-токене/XSS. **Фикс:** не возвращать пароль (или только `hasPassword`), шифровать at-rest, `select:false` + `@Exclude`.

### H6. HTML/контент-инъекция в исходящих письмах
- **Security** · `email.service.ts:91-105,133-143`; `email-templates.ts:56-61,126-131`
- `fullName`, `email`, `temporaryPassword`, ссылка сброса подставляются в HTML без экранирования (в отличие от `contract-approval-notification.service.ts`, где `escapeHtml` применяется правильно). **Фикс:** переиспользовать `escapeHtml` для всех подстановок.

### H7. Широкое раскрытие данных всем WebSocket-клиентам
- **Security** · `websocket.service.ts:269-286,359-395`
- Ролево ограничены только `notes:unread-refresh` и `contract-approval:updated`. События `plan:updated` (с полным объектом `MonthlyPlan`), `plan:recalculated`, `plan:admin-override`, `planning-v2:segment-updated`, `financial-plan:updated` рассылаются всем аутентифицированным сокетам без проверки сегмента/роли. Низкопривилегированный пользователь получает финансовые данные, недоступные ему через REST. **Фикс:** авторизация в `broadcastPlanUpdate` по аналогии с `PlanningV2Service.canViewSegment`.

### H8. Токен в localStorage (кража при XSS)
- **Security** · `frontend/src/store/auth-store.ts:23-53` (zustand `persist`); чтение в `services/api.ts:19-22`, `websocket-url.ts`, `warehouse-tus-upload.ts:41`
- Bearer-токен и объект пользователя persist-ятся в `localStorage`, читаемы любым скриптом на origin. **Фикс:** httpOnly+Secure+SameSite cookie либо токен только в памяти + refresh-cookie; строгий CSP.

### H9. Токен WebSocket передаётся в query string
- **Security** · `frontend/src/services/websocket-url.ts:25-27` (и `plans-realtime.ts:38-45`); зеркально backend `websocket.service.ts:201-215`
- `?token=` попадает в логи прокси, APM, историю браузера. **Фикс:** аутентификация handshake через cookie / `Sec-WebSocket-Protocol` / одноразовый тикет.

### H10. Авторизация только на клиенте (роль из редактируемого состояния)
- **Security** · `components/auth/RouteAccessGuard.tsx:9-15`, `routes/appRoutes.tsx:56-164`, `utils/rolePermissions.ts`
- Доступ к маршрутам/меню/редактированию считается из `user.role`, который лежит в `localStorage` и правится пользователем. Реальная брешь — только если backend не перепроверяет. **Фикс:** трактовать клиентские проверки как UX; убедиться, что каждый backend-эндпоинт проверяет роль/владение по JWT-клеймам (см. также H1, H7).

### H11. Fire-and-forget в Excel-импорте портит результат и порядок пересчёта
- **Bug** · `excel-import-export.service.ts:154-198`
- Внутри синхронного `worksheet.eachRow` вызывается `savePlan(...)` без `await`; счётчики/ошибки в оторванных `.then/.catch`. Итог: `importedCount`/`errors` почти всегда неверны, `recalculateAllPlans(year)` часто выполняется до сохранений (или не выполняется), ошибки становятся unhandled rejection. **Фикс:** собрать строки, обработать в `for...of` с `await` (лучше в одной транзакции), пересчёт — после.

## 6. Положительные наблюдения (сделано верно)
- bcrypt cost 12; `forgotPassword` без enumeration; reset-токен подписан `secret + passwordHash` (авто-инвалидация при смене пароля).
- IDOR-защита в контрактах/складе/планировании/биллинге; проверка «действующий пользователь = назначенный аппрувер».
- Нет SQL-инъекций: сырые запросы используют захардкоженные идентификаторы + `$1/$2`; миграции — статический DDL.
- docxtemplater/PizZip: экранирование и allowlist плейсхолдеров — инъекция шаблонов митигирована.
- Внешние хосты (FNS, Sinokor) захардкожены + `encodeURIComponent` — нет SSRF.
- `execFile` LibreOffice — фиксированный массив аргументов + whitelist бинаря (без shell).
- Секреты/логи/uploads/backups/xlsx/dist исключены и из git, и из docker-образа; в логах не найдено утёкших токенов/паролей.
- `content-disposition.ts` вычищает CR/LF и RFC5987-кодирует имя — нет header-инъекции.
- Фронт: нет `dangerouslySetInnerHTML`/`eval`; `safeCalculator.ts` — настоящий токенайзер/RPN без eval; защита от open-redirect на логине.
